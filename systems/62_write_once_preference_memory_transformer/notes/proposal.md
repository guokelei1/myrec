# C62 proposal — write-once preference-memory Transformer

Status: pre-outcome design.  No C62 model outcome or fresh label has been
observed.

## Observation → architecture consequence → falsification

**Observation.** C56--C60 show that history can affect candidates and that a
strong base can be protected, but fixed semantic evidence remains
misdirected.  C61 then trained a factual-minus-NULL edge Transformer with real
fit labels; its likelihood branch had gradients and nonzero transports, yet
the learned scale was seed-dependent and crossed no ranking margin.  Earlier
query-conditioned attention, pooled memory, query transport, candidate graph,
and output-residual families also failed their nearest controls.  Re-selecting
raw history separately for every candidate therefore remains a plausible
source of unstable evidence.

**Architecture consequence.** C62 changes the Transformer state lifecycle.
It first forms a query-independent user state and only then permits conditional
ranking:

```text
H'       = HistoryTransformer(history)
M        = SlotWriteTransformer(learned_slots <- H')
stopwrite(M)
X_i      = CandidateStem(query, candidate_i, base_coordinate_i)
R_i      = ReadTransformer(X_i <- M)
X'_1..C  = CandidateSetTransformer(R_1..C)
score_i  = base_i + centered_head(X'_i)
```

`stopwrite` is an attention-graph constraint, not gradient detachment: ranking
gradients train the history encoder and slot writer, but query/candidate states
cannot modify the already formed memory during the forward pass.  The same
memory is reused for every candidate and is exactly invariant to query or
candidate substitution.  Multi-slot memory can retain several user interests;
the later joint query-candidate read selects among them.

No offline memory, prompt scorer, fixed-score router, user-ID table, or MLP-only
ranker is present.  The learned residual is generated by candidate states after
both Transformer phases.  Empty history returns the base exactly, and
repeat-present requests retain the registered item-only fallback.

**Falsification.** The primitive is rejected if any of the following occurs:

1. memory changes when only query/candidates change, or query/candidates can
   write into the memory phase;
2. empty-history, repeat, candidate-permutation, determinism, or finite-gradient
   contracts fail;
3. deleting all but one slot does not materially reduce a planted multi-
   preference synthetic task;
4. on exposed fit-holdout roles, it does not beat the strong base, wrong
   history, and the three matched nearest controls in both KuaiSearch and
   Amazon-C4;
5. fresh label-free scoring is inactive or insensitive to wrong history.

There is no slot-count, width, scale, epoch, seed, subset, query-type, or domain
specific rescue after a frozen gate fails.

## One primitive and four matched modes

All modes instantiate the same parameters and optimization path.

- `write_once_memory` (primary): history-only multi-slot write, immutable
  query-candidate read.
- `query_conditioned_writer`: adds the query to slot formation, testing whether
  write/read separation matters.
- `direct_history_attention`: candidate tokens read encoded history directly,
  testing whether latent memory is merely renamed cross-attention.
- `single_pooled_slot`: all written slots are pooled and repeated before the
  read, testing whether multiple preference states are load-bearing.

A history-free forward using the same checkpoint is also scored as a
counterfactual control.  Parameter count, initialization family, training
labels, loss, optimizer, steps, candidate-set block, and output head remain
matched.

## Same-domain and cross-domain boundary

The operator, slot count, hidden width, losses, activity thresholds, and
decision rules are identical in both domains.  Only the frozen input embedding
width and registered seeds differ.

- KuaiSearch exposed training: C26 fit (6,000 requests); fresh A: C26
  internal-A (1,200, labels still closed).
- Amazon-C4 exposed training: C38 fit (6,000 requests); fresh source: the 399
  never-materialized C39 reserve, deterministically split 300 A / 99 delayed.

The first real gate uses only a deterministic 4,800/1,200 split inside the
already exposed fit roles.  Failure there closes C62 without materializing or
opening either fresh role.  Dev, test, and qrels are never authorized by this
proposal.

## Why this is problem-aligned rather than dataset-aligned

The architecture consumes only `(query, ordered prior history, candidates,
history_present, repeat_present)` and the registered non-personalized base
coordinate.  It contains no dataset identity, taxonomy, language, category,
history-length rule, score threshold, or top-k special case.  Its hypothesis is
about causal information flow: a stable user representation must be formed
before the current decision can interrogate it.

## Predicted failure modes

- slots collapse to one pooled state, so the single-slot control ties;
- direct history cross-attention learns the same function more easily;
- query-independent writing discards the query-specific evidence needed for
  product search;
- the learned residual either remains below base margins or overwrites the
  strong KuaiSearch base;
- wrong-user history is internally coherent and remains useful;
- Amazon improves only because its base is weaker, while KuaiSearch does not.

Any one is informative and terminal under the fixed gate.
